# 项目名称：AI Parallel Chat (平行衍生交互系统)

## 1. 核心概念与命名
* **功能名称**：**平行衍生 (Parallel Derivation)**
* **模式开关**：**衍生开关 (Derive Toggle)**
* **核心逻辑**：在主对话流不滚动、不更新的前提下，开启一个镜像/分支窗口，允许用户针对当前主窗口的某一部分内容进行深入追问，并自主选择是否将追问的认知“同步”回主记忆轴。

---

## 2. 系统架构设计



### 技术栈建议
* **前端 (Frontend)**：**React.js** (利用其强大的状态管理) + **Tailwind CSS** (实现分屏响应式布局) + **Lucide-React** (图标)。
* **后端 (Backend)**：**Python FastAPI**。Python 是 AI 生态的主流语言，方便集成各类 SDK。
* **数据库**：**PostgreSQL** (存储长对话) + **Redis** (缓存临时分支记忆)。
* **大模型调度**：使用 **LiteLLM** 库（一个统一的 Python 库，可以直接调用 OpenAI, Claude, Gemini, DashScope 等，无需为每个模型写适配器）。

## 3. 功能模块详细表述

### 3.1 模型配置模块 (Provider Manager)
* **功能描述**：支持用户录入并管理多个模型的 API Key。
* **支持列表**：包含 DashScope (通义千问)、DeepSeek、OpenAI、Gemini、Azure OpenAI。
* **配置项**：Base URL, API Key, Model Name, Temperature, Max Tokens。

### 3.2 交互窗口逻辑 (Dual-Pane UI)
* **主窗口 (Main Thread)**：默认全屏或左侧显示。展示完整的线性对话。
* **衍生窗口 (Parallel Pane)**：
    * 当“衍生开关”关闭时，该窗口隐藏。
    * 当“衍生开关”开启时，主窗口收缩至左侧（占 50%-60% 宽度），右侧滑出衍生窗口。
    * **上下文继承**：开启瞬间，衍生窗口自动拷贝一份当前主窗口的所有对话历史作为“基础背景”。

### 3.3 “平行衍生”工作流 (The Core Workflow)
1.  **触发**：用户在阅读主窗口回答时，点击“开启衍生”。
2.  **提问**：用户在右侧窗口输入问题。
3.  **处理**：后端接收请求，将“主窗口历史 + 右侧衍生过程”封装发送给模型。
4.  **记忆控制开关**：
    * **记忆同步 (Sync Memory)**：若开启，衍生窗口的所有问答在关闭模式后，会以“隐藏上下文”的形式合并进主对话的历史记录中。下次在主窗口提问，模型会记得衍生窗口的内容。
    * **纯净模式 (Pure Mode)**：若关闭，衍生窗口的内容仅作为一次性探讨，关闭后即销毁，不影响主窗口的后续交互。

### 3.4 记忆合并机制 (Memory Merging)
* **技术实现**：
    * 在数据库中，每条消息增加一个字段 `thread_id`（主线程）和 `parent_id`。
    * 当选择“加入记忆”并回到主窗口时，系统将分支对话标记为 `is_hidden: true`，但在调用 API 构造 `messages` 数组时，依然按时间顺序将这些隐藏消息插入其中。
    * **前端表现**：主窗口界面保持整洁，不显示分支对话的文字，但模型已经“读”过了。

## 5. 开发建议（给实现者的提示）

1.  **状态同步是难点**：建议使用 `Zustand` 或 `Redux` 来管理 `isParallelMode` 状态。当状态改变时，主窗口的消息流不应重新渲染（防止丢失滚动位置）。
2.  **样式细节**：建议在衍生窗口使用不同的背景色（如深一号的灰色），以便用户一眼分辨出“我是在分支中提问”。
3.  **Prompt 策略**：进入衍生模式提问时，可以自动在 Prompt 前缀加上“*基于左侧讨论的内容，我有以下具体的追问...*”，以增强模型的关联能力。
